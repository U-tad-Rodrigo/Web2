T1 · Node.js Fundamentos

¿Qué es Node.js?

Node.js es un runtime de JavaScript en el servidor basado en el motor V8 de Chrome + la biblioteca libuv. Lo que lo hace único: es single-threaded pero maneja I/O asincrónico de forma no bloqueante gracias al Event Loop.

V8 Engine

Compila JS a código máquina (JIT). Gestiona el heap de memoria y el call stack.

libuv

Librería C que implementa el Event Loop y el thread pool para I/O. Abstrae el sistema operativo.

Event Loop

Bucle infinito que decide qué ejecutar a continuación cuando el call stack está vacío.

Thread Pool

4 threads por defecto para operaciones costosas (fs, crypto, DNS). No bloquea el event loop.

El event loop tiene 6 fases que se ejecutan en orden. Entre cada fase se vacía la microtask queue (Promise.then, process.nextTick).

Orden de ejecución — Ejemplo clave

console.log('1');                    // síncrono → primero
setTimeout(() => console.log('2'), 0); // macrotask → último
Promise.resolve().then(() => console.log('3')); // microtask
process.nextTick(() => console.log('4'));  // microtask prioritaria
console.log('5');                    // síncrono → segundo
// Resultado: 1, 5, 4, 3, 2
En examen siempre preguntan el orden de ejecución. Regla: síncrono → nextTick → Promise.then → setTimeout/setImmediate.

ESM vs CommonJS

CommonJS (viejo)
const fs = require('fs')
module.exports = { fn }
ESM (moderno)
import fs from 'node:fs/promises'
export { fn }
export default fn
Para usar ESM: añadir "type": "module" en package.json o usar extensión .mjs. ESM soporta top-level await.

async/await — La forma moderna

async function readFile(path) {
  try {
    const data = await fs.readFile(path, 'utf-8');
    return data;
  } catch (err) {
    throw new Error(`No se pudo leer: ${err.message}`);
  }
}
// await es azúcar sintáctico sobre Promises
// SIEMPRE usa try/catch con await
async/await NO hace el código síncrono. Sigue siendo asincrónico — solo parece síncrono visualmente. Una función async SIEMPRE devuelve una Promise.

node:path

node:crypto

randomUUID(), createHash('sha256'), randomBytes(). Para tokens y hashes.

node:http

process.env, process.argv, process.exit(), process.cwd()

npm semver

^1.2.3 = minor+patch, ~1.2.3 = solo patch, 1.2.3 = exacta

Preguntas de Defensa — T1

¿Por qué Node.js puede manejar miles de conexiones con un solo hilo?▶
Porque usa I/O no bloqueante. Cuando Node hace una operación de I/O (leer archivo, consulta BD), en lugar de esperar bloquea el hilo, delega la operación al sistema operativo vía libuv y el hilo queda libre para atender otras peticiones. El Event Loop notifica cuando el I/O completó. Esto funciona perfectamente para I/O-bound workloads (servidores web, APIs).
¿Cuándo NO es adecuado Node.js?▶
Para tareas CPU-intensivas: compresión de video, cálculos matemáticos pesados, ML inference. Estas bloquean el hilo principal porque no hay I/O — simplemente calculan. Alternativas: usar worker_threads, delegar a microservicios en Python/Rust, o usar worker processes.
¿Cómo cambiarías el servidor para procesar CPUs sin bloquear el Event Loop?▶
Usaría worker_threads de Node.js para ejecutar la tarea CPU en un hilo separado: const { Worker } = require('worker_threads'). El worker procesa la tarea pesada y envía el resultado de vuelta al hilo principal via messagePassing. El Event Loop nunca se bloquea.
¿Qué diferencia hay entre require() e import?▶
require() (CommonJS) es síncrono y resuelve módulos en tiempo de ejecución. import (ESM) es asíncrono, se resuelve en tiempo de parsing (estático), lo que permite tree-shaking y análisis estático. ESM soporta top-level await. require() NO soporta await a nivel de módulo. La recomendación actual es ESM para proyectos nuevos.
Si en la defensa preguntan "¿cómo optimizarías el servidor?", siempre menciona: clustering (usar todos los cores CPU con pm2 cluster mode), caching en memoria (Redis), y connection pooling en la BD.

Promise Combinators — cuándo usar cada uno

MétodoCuándo usarloSi uno falla...
Promise.all(arr)Todas deben cumplirse. Si una falla, cancela todo.Rechaza inmediatamente. Las otras siguen ejecutándose pero se ignora resultado.
Promise.allSettled(arr)Quieres resultados de TODAS, fallen o no.Espera a todas. Retorna array de {status, value/reason}.
Promise.race(arr)Quieres la más rápida (timeout pattern).Si la primera es rechazo, rechaza.
Promise.any(arr)Quieres la primera que se cumpla.Solo rechaza si TODAS fallan (AggregateError).
// Patrón timeout con Promise.race
const withTimeout = (promise, ms) =>
  Promise.race([
    promise,
    new Promise((_, reject) =>
      setTimeout(() => reject(new Error('Timeout')), ms)
    )
  ]);

// Cargar múltiples recursos en paralelo
const [user, posts] = await Promise.all([
  getUser(id),
  getPosts(id)
]);

EventEmitter

import { EventEmitter } from 'node:events';

class OrderQueue extends EventEmitter {
  add(order) {
    this.emit('order:new', order);  // dispara evento
  }
}

const queue = new OrderQueue();
queue.on('order:new', (order) => { ... });  // suscripción permanente
queue.once('order:new', (order) => { ... }); // solo primera vez
queue.off('order:new', handler);            // desuscribir
Los listeners de EventEmitter son síncronos por defecto. Si el handler lanza un error y no hay listener de 'error', el proceso termina. Siempre añadir: emitter.on('error', handler).

process.nextTick vs setImmediate

process.nextTick(): se ejecuta al final del código síncrono actual, ANTES de que el event loop continúe. Tiene prioridad sobre Promise.then.

setImmediate(): se ejecuta en la fase CHECK del event loop, después de la fase poll. Puede ejecutarse después o antes que setTimeout(fn,0) dependiendo del contexto.

Async Iterators

// Leer stream línea por línea (Node.js 18+)
import { createReadStream } from 'node:fs';
import readline from 'node:readline';

const rl = readline.createInterface({ input: createReadStream('file.txt') });
for await (const line of rl) {
  processLine(line);  // procesa cada línea sin cargar todo en memoria
}
for await...of es la forma idiomática de consumir streams en Node.js moderno. Úsalo cuando necesites procesar grandes archivos sin cargarlos en memoria.

Preguntas de Defensa — T2

¿Cuándo usarías Promise.allSettled en vez de Promise.all?▶
Cuando necesitas procesar TODOS los resultados aunque alguno falle. Ejemplo: enviar notificaciones a múltiples emails — si uno falla no quieres cancelar los demás. Con allSettled obtienes array de {status:'fulfilled', value} o {status:'rejected', reason} para cada promesa, puedes procesar errores individualmente.
¿Cómo implementarías retry con backoff exponencial?▶
async function retry(fn, maxAttempts = 3) {
  for (let i = 0; i < maxAttempts; i++) {
    try { return await fn(); }
    catch (err) {
      if (i === maxAttempts - 1) throw err;
      await new Promise(r => setTimeout(r, 2 ** i * 1000)); // 1s, 2s, 4s
    }
  }
}
¿Qué es un memory leak con EventEmitter y cómo lo evitas?▶
Si añades listeners con .on() dentro de loops o funciones que se llaman repetidamente, acumulas listeners que nunca se limpian. Node.js avisa con "MaxListenersExceededWarning" (límite default: 10). Soluciones: usar .once() si solo necesitas una ejecución, siempre guardar referencia al handler para poder hacer .off(), o aumentar el límite con emitter.setMaxListeners(20).

PUT es idempotente porque si lo llamas 3 veces con los mismos datos, el resultado es el mismo. PATCH puede no serlo: "incrementar precio en 10" aplicado 3 veces da resultado diferente.

La diferencia entre 401 y 403: 401 = "no sé quién eres" (no autenticado), 403 = "sé quién eres pero no tienes permiso". Esta distinción siempre cae en examen.

¿Cuándo usarías PUT vs PATCH?▶
PUT para reemplazar el recurso completo (debes enviar todos los campos). Si el recurso no existe, lo crea. PATCH para actualizaciones parciales (envías solo los campos que cambian). En la práctica, PATCH es más común en APIs REST modernas porque el cliente no necesita conocer el estado completo del recurso.
¿Cómo implementarías rate limiting en tu API?▶
Con el middleware express-rate-limit: define un límite (ej: 100 requests por 15 minutos por IP). Para APIs de login, aplico un límite más estricto (5 intentos por 15 min) para prevenir brute-force. Puede usar Redis como store para distribuir el rate limit entre múltiples instancias del servidor. Retorna 429 Too Many Requests cuando se supera el límite.
¿Qué es idempotencia y por qué importa en una API REST?▶
Una operación es idempotente si ejecutarla N veces tiene el mismo efecto que ejecutarla 1 vez. Importa para la tolerancia a fallos: si un cliente no sabe si su petición llegó, puede reenviarla de forma segura si es idempotente (GET, PUT, DELETE). POST no es idempotente — reenviar podría crear duplicados. Por eso POST de pago debe tener una clave de idempotencia única.

Express 5 vs Express 4 — Lo nuevo

Express 4
// Async errors NO capturados
app.get('/user', async (req, res) => {
  const user = await getUser(); // Si falla
  // El error NO llega al error handler
  // → crash del proceso
})
Express 5 (automático)
// Async errors SÍ capturados automáticamente
app.get('/user', async (req, res) => {
  const user = await getUser(); // Si falla
  // Express 5 captura y pasa al error handler
  // → respuesta controlada
})
La diferencia CLAVE de Express 5: manejo automático de errores en funciones async. En Express 4 tenías que usar try/catch + next(err) manualmente en cada handler.

// Middleware típico: 3 parámetros = middleware normal
const authMiddleware = async (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'No token' });
  req.user = await verifyToken(token);
  next(); // pasa al siguiente middleware
};

// Error handler: 4 parámetros (err, req, res, next) — SIEMPRE al final
app.use((err, req, res, next) => {
  console.error(err);
  res.status(err.status || 500).json({ error: err.message });
});
El error handler DEBE tener exactamente 4 parámetros (err, req, res, next) aunque no uses next. Si pones 3, Express no lo reconocerá como error handler.

Estructura de Proyecto MVC

src/
├── app.js              ← setup Express, middlewares globales
├── config/             ← db.js, swagger.js, env.js
├── routes/             ← user.routes.js, project.routes.js
├── controllers/        ← user.controller.js (lógica HTTP)
├── services/           ← user.service.js (lógica de negocio)
├── models/             ← user.model.js (schemas Mongoose/Prisma)
├── middlewares/        ← auth.js, validate.js, upload.js
└── utils/              ← errors.js, jwt.js, email.js
Regla: Controllers solo hablan HTTP (req, res). Services son la lógica de negocio, no conocen Express. Esto hace los servicios testables sin HTTP.

Express 5 — Sintaxis nueva en rutas

Express 4
app.get('/:id?', handler)  // param opcional
app.get('/*', handler)     // wildcard
Express 5
app.get('/{:id}', handler)  // param opcional
app.get('/*filepath', handler) // wildcard nombrado
¿Por qué separar Controller de Service?▶
El Controller conoce Express (req, res, next). El Service no conoce HTTP — solo recibe datos y devuelve resultados. Ventajas: (1) Puedes testear el Service sin arrancar un servidor HTTP. (2) Puedes reutilizar el Service desde CLI, jobs, otras rutas. (3) Si migras de Express a Fastify, solo cambias los controllers. El código de negocio no se toca.
¿Cómo añadirías logging a todas las peticiones sin modificar cada route?▶
Con un middleware global registrado antes que las rutas: app.use((req, res, next) => { console.log(req.method, req.url); next(); }). O usando morgan o pino-http que ya lo hacen bien. Se registra una sola vez y aplica a TODAS las rutas automáticamente.
¿Cómo manejarías distintos tipos de errores (validación, auth, bd) en el error handler?▶
Creo clases de error personalizadas: class ValidationError extends Error { constructor(msg) { super(msg); this.status = 400; } } y class AuthError extends Error { this.status = 401 }. El error handler central comprueba el tipo: if (err instanceof ValidationError) return res.status(400).json(...). Así el error handler es genérico y cada error lleva su propio status.

Schema con tipos y validación

const userSchema = new Schema({
  name:    { type: String, required: true, trim: true, minlength: 2 },
  email:   { type: String, required: true, unique: true, lowercase: true },
  role:    { type: String, enum: ['ADMIN', 'USER'], default: 'USER' },
  company: { type: Schema.Types.ObjectId, ref: 'Company' }, // referencia
  deleted:    { type: Boolean, default: false },           // soft delete
  deletedAt:  { type: Date },
}, { timestamps: true });  // añade createdAt y updatedAt

Soft Delete Pattern — MUY IMPORTANTE

En lugar de borrar físicamente los documentos, los marcamos como eliminados. Esto permite auditoría, recuperación, y evita referencias rotas.

// DELETE → marcar como deleted, no borrar físicamente
async deleteUser(id, deletedBy) {
  return User.findByIdAndUpdate(id, {
    deleted: true,
    deletedAt: new Date(),
    deletedBy
  });
}

// SIEMPRE filtrar deleted:false en queries
const users = await User.find({ deleted: false });

// Con plugin (mejor): automático en todos los modelos
userSchema.plugin(softDeletePlugin);
El soft delete es un patrón fundamental. En BildyApp TODOS los modelos lo implementan. Siempre añadir deleted:false a los queries para no mostrar datos borrados.

Population — Relaciones entre documentos

// populate() reemplaza el ObjectId con el documento completo
const projects = await Project
  .find({ deleted: false })
  .populate('client', 'name email')  // solo name y email del cliente
  .lean();  // retorna POJOs (más rápido, no tiene métodos Mongoose)

// Embedded vs References:
// Embedded: datos dentro del documento (rápido, pero duplica datos)
// References: ObjectId → necesita populate (más flexible)

Middleware Hooks pre/post

// pre-save: hashear password antes de guardar
userSchema.pre('save', async function() {
  if (!this.isModified('password')) return;
  this.password = await bcrypt.hash(this.password, 12);
});

// post-save: enviar email de bienvenida
userSchema.post('save', async function(doc) {
  await sendWelcomeEmail(doc.email);
});
Dentro de un hook pre('save'), usa this para acceder al documento. Pero si usas findByIdAndUpdate en lugar de save, los hooks de 'save' NO se disparan. Usa hooks de 'findOneAndUpdate' si quieres interceptar updates.

Indexes para Performance

// Índice único (email no puede repetirse)
email: { type: String, unique: true }

// Índice compuesto (queries que filtran por company Y deleted)
schema.index({ company: 1, deleted: 1 });

// Índice de texto para búsqueda full-text
schema.index({ name: 'text', description: 'text' });
¿Cuándo usarías embedded documents vs references en MongoDB?▶
Embedded (datos dentro del documento): cuando los datos siempre se acceden juntos, no van a crecer indefinidamente, y son propios del documento. Ej: array de tags, dirección de un usuario. References (ObjectId): cuando los datos se comparten entre múltiples documentos, crecen independientemente o se consultan por separado. Ej: user → company, project → client. MongoDB no tiene JOINs nativos, pero populate() simula el comportamiento.
¿Cómo harías el borrado en cascada? Si elimino una Company, se eliminan sus Projects.▶
Con un hook post('save') o pre('findOneAndUpdate') en el modelo Company que detecte cuando deleted cambia a true, y marque todos los proyectos relacionados como deleted también. También se puede hacer en el service con una transacción de MongoDB (session.withTransaction). El soft delete en cascada es más seguro porque todo queda auditado.
¿Qué es lean() y cuándo usarlo?▶
.lean() hace que Mongoose devuelva objetos JavaScript planos (POJO) en lugar de instancias de Mongoose con todos sus métodos (.save(), .populate(), etc). Ventajas: más rápido y menos memoria. Usarlo cuando solo necesitas leer datos y no necesitas guardar cambios ni usar métodos de Mongoose en los documentos. NO usar lean() si vas a modificar y guardar el documento.

¿Por qué Zod?

Zod valida en tiempo de ejecución y genera tipos TypeScript automáticamente. Es declarativo, composable, y permite transformaciones.

Schemas básicos

import { z } from 'zod';

const CreateUserSchema = z.object({
  name:     z.string().min(2).max(100).trim(),
  email:    z.string().email().toLowerCase(),
  password: z.string().min(8).regex(/[A-Z]/, 'Necesita mayúscula'),
  age:      z.number().int().min(18).max(120).optional(),
  role:     z.enum(['ADMIN', 'USER']).default('USER'),
});

// Inferir tipo TypeScript automáticamente
type CreateUser = z.infer<typeof CreateUserSchema>;

Refine y SuperRefine — Validación custom

// .refine() para 1 sola validación custom
const PasswordSchema = z.object({
  password: z.string().min(8),
  confirm:  z.string(),
}).refine((data) => data.password === data.confirm, {
  message: 'Las contraseñas no coinciden',
  path: ['confirm'],  // en qué campo aparece el error
});

// .superRefine() para múltiples errores custom
.superRefine((data, ctx) => {
  if (!/[0-9]/.test(data.password))
    ctx.addIssue({ code: z.ZodIssueCode.custom, message: 'Necesita número', path: ['password'] });
  if (!/[!@#$]/.test(data.password))
    ctx.addIssue({ code: z.ZodIssueCode.custom, message: 'Necesita símbolo', path: ['password'] });
});
refine = 1 error, superRefine = múltiples errores. En superRefine usas ctx.addIssue() para añadir cada error individualmente.

Composition de Schemas

const BaseSchema = z.object({ name: z.string(), email: z.string().email() });

Middleware de validación en Express

const validate = (schema) => (req, res, next) => {
  const result = schema.safeParse(req.body);
  if (!result.success) {
    return res.status(400).json({ errors: result.error.flatten().fieldErrors });
  }
  req.body = result.data;  // datos ya validados Y transformados
  next();
};

// Uso en ruta:
router.post('/users', validate(CreateUserSchema), userController.create);
¿Por qué validar datos aunque la BD también tenga validaciones?▶
Defense in depth. La validación en la BD es la última línea de defensa, pero es mejor fallar rápido en la capa de aplicación (Zod) antes de hacer un round-trip a la BD. Además, Zod puede dar mensajes de error más descriptivos y específicos para el usuario. También, Zod puede transformar datos (trim, toLowerCase) antes de guardarlos, lo que la BD no hace automáticamente.
¿Cómo validarías que dos fechas son coherentes (startDate < endDate)?▶
Con .refine() en el objeto completo (no en un campo individual): z.object({startDate: z.date(), endDate: z.date()}).refine(d => d.startDate < d.endDate, {message: 'endDate debe ser posterior a startDate', path: ['endDate']}). El path indica en qué campo aparece el error en el cliente.

JWT — Estructura

Un JWT tiene 3 partes separadas por puntos, codificadas en Base64URL (no encriptadas, solo firmadas):

HEADER
alg: HS256, typ: JWT
.
PAYLOAD
userId, role, exp
.
SIGNATURE
HMAC(header+payload, secret)
El payload de un JWT es visible por cualquiera (solo está en Base64, no encriptado). NUNCA pongas información sensible (password, datos bancarios) en el payload. La firma garantiza integridad, no privacidad.

bcrypt — Hash de passwords

import bcrypt from 'bcryptjs';

// Hash (registro): bcrypt genera salt automáticamente

// Verificar (login): NO son iguales, bcrypt compara internamente
const match = await bcrypt.compare(plaintext, hashed); // true/false

// NUNCA: hashed === hashed2  (mismo password → hashes DISTINTOS por el salt)
Salt rounds: cada incremento de 1 duplica el tiempo de cómputo. 10 rounds = ~100ms, 12 rounds = ~300ms. Recomendado: 10-12. Más rounds = más seguro pero más lento el login.

Refresh Token Rotation

Cada vez que usas el refresh token, recibes un NUEVO refresh token (además del nuevo access token). El anterior queda invalidado. Si alguien roba el refresh token y lo usa, el usuario original detectará que el suyo ya no funciona.

¿Cómo invalidarías todos los tokens de un usuario tras cambiar su contraseña?▶
Los JWT stateless no se pueden invalidar individualmente una vez emitidos. Opciones: (1) Guardar un campo tokenVersion en el usuario e incluirlo en el JWT. Al cambiar password, incrementar tokenVersion. El middleware verifica que la versión del token coincide. (2) Usar una blacklist en Redis con los tokens revocados. (3) Guardar los refresh tokens en BD y marcarlos como revocados. BildyApp usa el refresh token en BD para esto.
¿Por qué usar dos secrets distintos para access y refresh tokens?▶
Por seguridad. Si se compromete el secret del access token (más expuesto en producción), el atacante no puede generar refresh tokens. Además, permite rotar el secret del access token sin invalidar todos los refresh tokens. Son tokens con propósitos distintos: uno para autenticar peticiones (corta vida) y otro para renovar acceso (larga vida).
¿Cómo cambiarías el sistema para implementar login con Google OAuth?▶
Usaría passport.js con passport-google-oauth20. El flujo: (1) Usuario hace clic en "Login con Google", redirige a Google. (2) Google autentica y redirige a /auth/google/callback con un code. (3) El servidor intercambia el code por un token de Google, obtiene el perfil. (4) Busca o crea el usuario en BD. (5) Genera los JWT propios de la app y devuelve al frontend. El usuario no necesita password.

¿Cuándo usarías Prisma (SQL) en lugar de Mongoose (MongoDB)?▶
Prisma+PostgreSQL cuando: los datos son altamente relacionales (muchas relaciones entre entidades), necesitas ACID transactions fuertes (pagos, inventario), o el esquema es muy estable y predecible. Mongoose+MongoDB cuando: los datos son documentos flexibles con estructura variable, necesitas escalar horizontalmente con facilidad, o los datos son jerárquicos y raramente necesitas JOINs.
¿Qué pasaría si haces prisma migrate en producción con una migration que añade NOT NULL sin default?▶
Error de migración. Si tienes datos existentes y añades una columna NOT NULL sin valor default, PostgreSQL rechaza la migration porque las filas existentes no tendrían valor para esa columna. Solución: (1) Añadir primero la columna como nullable, (2) hacer un backfill (UPDATE) para rellenar los datos existentes, (3) luego cambiar a NOT NULL. Prisma migrate detecta esto y genera la migration en 3 pasos automáticamente.

WebSocket vs HTTP Polling

HTTP Polling (lento)
  • Cliente pregunta cada N segundos
  • Alta latencia
  • Muchas peticiones innecesarias
  • Overhead de headers HTTP
WebSocket (tiempo real)
  • Conexión persistente bidireccional
  • Latencia mínima (~1ms)
  • Servidor PUSH datos cuando hay novedad
  • Sin overhead HTTP por mensaje

Socket.IO — Broadcast Patterns

// Servidor
io.on('connection', (socket) => {
  // Solo al socket que se conectó
  socket.emit('welcome', { msg: 'Bienvenido' });

  // A todos excepto el emisor
  socket.broadcast.emit('user:joined', socket.id);

  // A todos (incluyendo el emisor)
  io.emit('notification', data);

  // Solo a una sala específica
  io.to(roomId).emit('message', data);

  // Unirse/salir de sala
  socket.join(roomId);
  socket.leave(roomId);

  // Escuchar evento del cliente
  socket.on('message:send', async (data) => {
    const msg = await saveMessage(data);
    io.to(data.roomId).emit('message:new', msg);
  });
});

Autenticación en Socket.IO

// Middleware en el handshake (antes de conectar)
io.use(async (socket, next) => {
  try {
    const token = socket.handshake.auth.token;
    socket.user = jwt.verify(token, ACCESS_SECRET);
    next();
  } catch {
    next(new Error('Unauthorized'));
  }
});

// Cliente: envía el token en el handshake
const socket = io('http://localhost:3000', {
  auth: { token: localStorage.getItem('token') }
});

Namespaces vs Rooms

Namespaces

Endpoints separados: /chat, /admin, /notifications. Separan funcionalidades a nivel de conexión. io.of('/chat')

Rooms

Grupos dentro de un namespace. Un socket puede estar en múltiples rooms. socket.join('room:123'). Ideal para chats de grupo, notificaciones por empresa.

En BildyApp: cada empresa tiene su propia room. Cuando hay un cambio en un proyecto, se emite a company:{companyId} — todos los usuarios de esa empresa reciben la notificación en tiempo real.
¿Cómo escalarías Socket.IO a múltiples servidores?▶
Con @socket.io/redis-adapter. El problema: si tienes 3 servidores y un cliente se conecta al servidor 1, pero el evento debe llegar a clientes en el servidor 2 y 3. Redis actúa como bus de mensajes: cuando el servidor 1 hace io.emit(), Redis lo publica y los servidores 2 y 3 lo reciben y lo envían a sus clientes conectados. Sin este adapter, socket.broadcast.emit() solo llegaría a clientes del mismo servidor.
¿Cómo manejarías la persistencia de mensajes si el usuario estaba offline?▶
Guardar todos los mensajes en MongoDB con campos roomId, senderId, content, timestamp, readBy[]. Cuando el usuario se conecta (connection event), consulta los mensajes no leídos desde su último visto. El cliente actualiza readBy[] cuando los muestra. Esta es la misma estrategia que WhatsApp: mensajes siempre persistidos, Socket.IO solo es el canal de entrega en tiempo real.

¿Cuál es la diferencia entre una imagen Docker y un contenedor?▶
Una imagen es una plantilla inmutable (como una clase en OOP). Un contenedor es una instancia en ejecución de esa imagen (como un objeto). Puedes tener múltiples contenedores ejecutándose de la misma imagen. La imagen se construye con docker build, el contenedor se crea con docker run.
¿Por qué usar named volumes en docker-compose y no bind mounts?▶
Named volumes (mongo_data:/data/db) son gestionados por Docker y persisten entre docker-compose down y docker-compose up. Bind mounts (./data:/data/db) montan un directorio del host — útil en desarrollo para hot-reload. Para bases de datos en producción, named volumes son mejores porque Docker los gestiona y tienen mejor rendimiento en Windows/Mac.

Docker/Kubernetes usan el health check para saber si el contenedor está listo para recibir tráfico y para auto-reiniciarlo si falla.

¿Cómo cambiarías el upload para que el usuario suba directamente a S3 sin pasar por tu servidor?▶
Con presigned URLs. El flujo: (1) Cliente pide un presigned URL a tu API (POST /uploads/presigned). (2) Tu servidor genera el URL con AWS SDK: const url = await getSignedUrl(s3, new PutObjectCommand({Bucket, Key}), {expiresIn: 300}). (3) El cliente usa ese URL temporal para subir el archivo directamente a S3 con PUT. (4) S3 guarda el archivo. (5) El cliente notifica a tu API que el upload completó. ¿Cómo prevendrías la subida de archivos maliciosos?▶
(1) Validar MIME type real con magic bytes (paquete file-type), no solo la extensión. (2) Renombrar el archivo con un UUID para evitar path traversal. (3) Límites de tamaño estrictos con Multer. (4) Almacenamiento fuera del servidor web (Cloudinary/S3) — el archivo no es ejecutable. (5) Si son documentos, scan de virus con ClamAV o una API de escaneo. (6) Para imágenes: procesarlas con Sharp (la imagen se re-renderiza, elimina metadatos y posibles exploits EXIF).
